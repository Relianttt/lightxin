# 查寝接口适配记录

## 文档目的

记录 `fdygl.aiit.edu.cn` 查寝签到链路在独立 App 中落地时已经踩到并确认过的适配点，供后续开发、重构和排障直接复用。

本文档基于 2026 年 4 月 15 日晚的实际联调结果整理，结论优先级高于早期规划中的静态假设。

## 当前已跑通范围

- 首页卡片进入查寝列表
- 查寝任务列表加载
- 查寝任务详情加载
- 拍照上传
- 定位签到提交
- 签到成功后返回列表并触发一次刷新

对应代码主入口：

## 1. 查寝接口鉴权不是只带 Bearer token

### 现象

初始只带：

- `Authorization: Bearer <accessToken>`

请求：

- `POST https://fdygl.aiit.edu.cn/app/dorm/pageStudentSignIn`

返回：

```json
{"flag":false,"code":"-100","msg":"非法访问","data":null}
```

### 最终确认可用的请求头

查寝 H5 实际请求除 Bearer token 外，还依赖以下身份头：

- `accessToken`
- `access_token`
- `userCode`
- `xh`
- `userType`
- `X-Requested-With: XMLHttpRequest`
- `Origin: https://fdygl.aiit.edu.cn`
- `Referer: https://fdygl.aiit.edu.cn/swp-app/`

### 代码位置

- `app/src/main/java/com/lightxin/core/network/AuthInterceptor.kt`

### 说明

后续如果有人重构网络层，不能把查寝域名简单并入“通用 Bearer token 模式”，否则列表接口会再次回到 `-100 非法访问`。

## 2. 查寝列表真实返回结构与早期假设不一致

### 实际成功返回

`pageStudentSignIn` 成功响应的关键结构是：

```json
{
  "code": "0",
  "flag": true,
  "data": null,
  "rows": [
    {
      "id": "2044075705918066690",
      "title": "23电信寝室晚点名签到任务(2026春季学期)",
      "executionedStatus": "0",
      "timedStartTime": "2026-04-15 21:00",
      "timedEndTime": "2026-04-15 23:30",
      "userTaskId": "2044075706840813696"
    }
  ],
  "total": 1
}
```

### 之前的错误假设

之前代码错误地按下面这套结构解析：

- `data.list`
- `taskName`
- `signinStatus`
- `startTime`
- `endTime`

这会导致接口明明返回了任务，页面却仍显示“暂无签到任务”。

### 正确映射建议

- 列表数组：优先读顶层 `rows`
- 任务标题：`title`
- 状态：`executionedStatus`
- 开始时间：`timedStartTime`
- 结束时间：`timedEndTime`
- 详情页入参：优先用 `id`，兼容 `taskDateId / userTaskId / taskId`

### 代码位置

- `app/src/main/java/com/lightxin/feature/checkin/data/CheckinResponse.kt`
- `app/src/main/java/com/lightxin/feature/checkin/data/CheckinRepository.kt`

## 3. 查寝详情接口字段也需要按真实返回解析

### 实际关键字段

`getTaskInfoByDateId` 返回里，真正有用的字段更接近：

- `title`
- `dateId`
- `signStatus`
- `photoRequire`
- `signinPhoto`
- `timedStartTime`
- `timedEndTime`
- `signinLocations`

其中 `signinLocations` 是重点：

```json
"signinLocations": [
  {
    "content": 500,
    "lngLat": "118.637378,31.141338",
    "signinLocation": "安徽信息工程学院(新芜校区)"
  }
]
```

### 解析规则

- 任务标题：`title`
- 任务日期 ID：`dateId`
- 是否已签到：`signStatus == "1"`
- 是否需要照片：`photoRequire == "1"` 或 `signinPhoto == "1"`
- 签到地点描述：优先 `signinLocations[0].signinLocation`
- 签到范围：优先 `signinLocations[0].content`
- 中心点：优先解析 `signinLocations[0].lngLat`

### 注意

不要只依赖：

- `taskName`
- `taskDateId`
- `needPhoto`
- `lng`
- `lat`
- `locationRange`

这些字段在当前接口里不是最稳定来源。

### 代码位置

- `app/src/main/java/com/lightxin/feature/checkin/data/CheckinResponse.kt`
- `app/src/main/java/com/lightxin/feature/checkin/data/CheckinRepository.kt`

## 4. 照片上传接口的 `data` 不是稳定字符串

### 现象

提交签到前的照片上传阶段曾出现：

```text
java.lang.IllegalStateException: Expected a string but was BEGIN_OBJECT at line 1 column 21 path $.data
```

### 根因

`uploadFileToFastdfs` 的响应 `data` 字段并不稳定，有时不是纯字符串，而是对象。

如果直接定义为：

```kotlin
val data: String?
```

Gson 会在反序列化阶段直接失败。

### 当前兼容策略

上传响应改为：

- `data: JsonElement?`

随后兼容提取以下常见字段：

- `url`
- `fileUrl`
- `fullPath`
- `path`
- `filePath`

如果没有这些显式键，再兜底扫描对象中的 URL 字符串。

### 代码位置

- `app/src/main/java/com/lightxin/feature/checkin/data/CheckinResponse.kt`
- `app/src/main/java/com/lightxin/feature/checkin/data/CheckinRepository.kt`

### 后续建议

如果后面抓到稳定上传返回样本，建议把当前“兼容提取”收敛成明确字段，减少动态兜底逻辑。

## 5. 列表状态不会自动刷新，需要显式回传刷新信号

### 现象

2026 年 4 月 15 日的联调中，签到成功后直接返回任务列表，状态不会立刻从“未签到”变成“已签到”；退出首页再重进后才更新。

### 根因

- 详情页成功后只做了返回
- 列表页 ViewModel 只在初始化时加载一次
- 返回时没有主动刷新

### 当前处理方式

通过导航返回结果做一次性刷新：

1. 详情页成功时给上一页写入 `checkin_refresh=true`
2. 列表页监听该标记
3. 收到后调用一次 `refresh()`
4. 立刻把标记重置为 `false`

### 优点

- 只在真正签到成功后刷新一次
- 不会每次返回列表都重打接口
- 改动集中在导航层和列表页，成本低

### 代码位置

- `app/src/main/java/com/lightxin/navigation/NavGraph.kt`
- `app/src/main/java/com/lightxin/feature/checkin/ui/CheckinListScreen.kt`
- `app/src/main/java/com/lightxin/feature/checkin/ui/CheckinViewModel.kt`
- `app/src/main/java/com/lightxin/feature/checkin/ui/CheckinDetailScreen.kt`

## 6. 首页卡片当前是查寝正式入口之一

### 现状

首页 `HomeDashboard` 中的“查寝签到”卡片已经绑定导航，直接进入查寝列表。

### 意义

当前查寝不是隐藏模块，首页入口已经成为正式用户路径。后续如果有人重构首页或导航，不能把这条入口当测试代码删掉。

### 代码位置

## 7. 仍建议后续补抓的内容

虽然当前链路已可用，但下面这些信息如果能补齐，会让实现更稳：

- `uploadFileToFastdfs` 的稳定返回 JSON 样本
- `signIn` 成功和失败时 `data` 的完整结构
- 查寝 H5 中是否还依赖 `role / roleId`
- 边界失败态：范围外、缺图、超时、重复签到

## 8. 给后续开发的直接建议

- 不要再按“规划里的理想 DTO”写查寝模型，优先以真实抓包和线上日志为准。
- 对查寝接口改动时，先保住以下三件事：
  - 鉴权头完整
  - 列表读 `rows`
  - 上传响应兼容对象
- 如果以后要做模块重构，优先保留 `Repository` 层的字段兼容逻辑，再考虑逐步收敛模型。
