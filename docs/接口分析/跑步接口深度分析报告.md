### 1.2 登录响应结构

```json
{
    "flag": true,
    "data": {
        "token": {
            "userCode": "学号/用户编码",
            "userName": "用户姓名",
            "userType": "用户类型(1=学生)",
            "expires_in": "有效期(秒)",
            "access_token": "访问令牌",
```

### 1.3 认证信息存储

登录成功后，以下字段存储到 `GlobalPreference`:

| 存储键 | 值来源 | 说明 |
|--------|--------|------|
| `access_token` | `data.token.access_token` | 访问令牌 |
| `refresh_token` | `data.token.refresh_token` | 刷新令牌 |
| `userCode` | `data.token.userCode` | 用户编码/学号 |
| `userName` | `data.token.userName` | 用户姓名（URL编码后参与请求） |
| `userType` | `data.token.userType` | 默认 `1`（学生） |
| `fileAddress` | `data.token.fileAddress` | 头像地址 |

- 新应用若要实现首页首屏，需要在登录完成后立即拉取上述接口

## 📚 4. 课程表模块

### 4.1 首页课程表预览接口

| 属性 | 值 |
|------|-----|
| URL | `https://in.aiit.edu.cn/zhxy-new-scps/appService/timeTable.do` |
| 方法 | POST |

**说明**:
- 该接口用于首页卡片预览，不是“课表查询”页面最终数据源
- 抓包返回示例字段：`classWeek`、`week`、`timeTableVo`
- `timeTableVo` 仅返回当天预览课程，例如 `虚拟仪器技术`、`习近平新时代中国特色社会主义思想概论`

### 4.2 课表查询入口前置请求（运行时确认）

从首页点击 `课表查询` 前，主站会先发起以下请求：

| URL | 方法 | 说明 |
|-----|------|------|
| `https://in.aiit.edu.cn/zhxy-new-scps/appService/checkServiceVersion.do` | POST | 服务版本校验 |
| `https://in.aiit.edu.cn/zhxy-new-scps/userLogin/click.do` | POST | 服务点击上报 |

**抓包实测参数**:
- `checkServiceVersion.do`
  - `serviceId=2c9095905e364af2015e3654f7710007`
  - 返回 `data="1"`
- `click.do`
  - `id=2c9095905e364af2015e3654f7710007`
  - `type=2`

### 4.3 课表查询 H5 入口（运行时确认）

| 属性 | 值 |
|------|-----|
| URL | `https://in.aiit.edu.cn/dist/` |
| 方法 | GET |
| 说明 | 课表查询实际 H5 容器页 |

**运行特征**:
- WebView 会继续加载 `/dist/static/js/...`、`/dist/static/css/...` 等静态资源
- 后续真实业务请求不再走 `appService/timeTable.do`，而是切换到 `zhxy-csh` 子系统
- 相关请求头带有：
  - `Origin: https://in.aiit.edu.cn`
  - `Referer: https://in.aiit.edu.cn/dist/`

### 4.4 周次列表接口（运行时确认）

| 属性 | 值 |

### 13.7 AI课堂扫码容器与返回主页异常（运行时确认）

**运行时现象**:
- `adb logcat` 显示，进入 AI课堂扫码流程时，应用会从 `WebActivity` 切到原生 `ScanAttendanceForWebViewActivity`
- 当前抓包尚未拿到“真实扫码成功后的业务提交请求”

**源码链路（已确认）**:
1. `WebActivity` 的 JSBridge `scanCode()` 拉起 `ScanAttendanceForWebViewActivity`
2. 原生扫码页识别二维码后，通过 `Intent.putExtra("url", 扫码结果)` 回传给 `WebActivity`
3. `WebActivity.onActivityResult(requestCode=0xb)` 调用 `sendScanToServer(url)`
4. `sendScanToServer(url)` 最终执行：

```javascript
javascript:scanCodeReturn("二维码内容")
```

**本次 logcat 异常证据（2026-04-13 22:20:12）**:
- `Failure delivering result ResultInfo{who=null, request=11, result=0, data=null}`
- `NullPointerException: Attempt to invoke virtual method 'java.lang.String android.content.Intent.getStringExtra(java.lang.String)' on a null object reference`
- `at cn.edu.aiit.axx.activity.WebActivity.onActivityResult(WebActivity.java:936)`
- `Force finishing activity cn.edu.aiit.axx/.activity.WebActivity`

**解释**:
- 当扫码页返回 `result=0` 且 `data=null` 时，`WebActivity.onActivityResult()` 没有做空值判断
- 这会直接打崩当前 `WebActivity`
- 这与用户体感里的“返回一下就回主页了”是吻合的，不只是主观感觉或普通路由跳转

**结论**:
- AI课堂扫码使用的是“原生扫码壳 + H5 消费二维码结果”的混合模式
- 原生层当前更像扫描器，不是最终业务提交者
- 真正的签到提交请求大概率仍在 FIF H5 页面内部触发
- 目前已经有足够证据确认：AI课堂至少存在一个扫码返回空数据导致 `WebActivity` 崩溃的稳定问题
