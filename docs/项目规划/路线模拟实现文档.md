# 轻小信 - 路线模拟实现文档

## 1. 文档目的

本文档用于细化 `正式版预期功能.md` 中“路线模拟后续功能”的落地方案，给出一套可以直接进入开发的实现路径。

目标不是一次性做完所有高级能力，而是先明确：

- 当前项目已有能力是什么
- 正式版路线模拟缺哪些关键模块
- 应该按什么顺序补齐
- 每个阶段最小要做到什么程度

## 2. 当前现状

结合当前代码，路线模拟链路已经具备 MVP 基础能力：

- 支持输入距离、时长、开始时间
- 支持生成模拟轨迹并走现有上传链路
- 支持真实跑步时采集轨迹点

当前实现的主要特点：

- 模拟页仅有基础参数输入，无路线模板选择与管理
- 轨迹生成器仍是“预设少量校园点 + 循环取点 + 轻微抖动”
- 真实跑步产生的轨迹仅用于本次上传，不会沉淀为可复用模板
- 没有设置中心、路线库、默认路线、质量校验等正式版能力

因此，当前状态可以定义为：

- `MVP 可用`
- `正式版路线模拟未落地`

## 3. 正式版目标

路线模拟正式版的核心目标不是“让轨迹更复杂”，而是让模拟轨迹具备以下三个特征：

- 更接近用户本人真实校园跑步习惯
- 可管理、可复用、可控
- 保持实现简单，避免过早引入复杂路线系统

本阶段建议聚焦以下能力：

- 在“我的”页新增路线模拟入口
- 支持在路线模拟设置页中独立录制模板
- 支持模板列表查看、重命名、删除、设为默认
- 模拟提交时优先使用默认模板生成轨迹
- 支持基于模板做截取、插值、拼接、轻微抖动
- 模板详情支持轨迹预览
- 保存模板前增加基础质量校验

## 4. 实现范围

### 4.1 本期要做

- “我的”页中的“路线模拟”入口
- 路线模拟设置页
- 本地路线模板存储
- 独立模板录制页
- 模板管理页
- 模板详情页
- 默认模板选择
- 基于模板的模拟轨迹生成
- 基础路线质量校验
- 模板详情轨迹预览

### 4.2 本期不做

- 云端同步路线模板
- 多校园地图可视化编辑
- 主跑步页地图化改造
- 复杂路线评分系统
- 机器学习式轨迹拟合
- 多模板智能混合推荐

## 5. 设计原则

- KISS 优先，先做本地可用
- 不改动现有上传加密与提交主链
- 不在真实跑步主链中混入模板管理能力
- 模板录制作为独立功能存在于设置体系
- 真实跑步、模板录制、模拟提交共用统一轨迹模型
- 模板管理尽量基于本地持久化，不引入额外后端依赖
- 算法先追求“稳定自然”，再追求“高保真”
- 录制页先保持数据仪表盘风格，地图能力优先放在模板详情预览

## 6. 整体方案

### 6.1 功能结构

建议增加以下结构：

- “我的”页入口
- 路线模拟设置页
- 模板录制页
- 路线模板列表页
- 路线模板详情页
- 模拟提交页中的模板状态区

业务链路如下：

1. 用户在“我的”页进入“路线模拟”
2. 进入路线模拟设置页
3. 用户选择“录制新模板”
4. 在独立录制页完成轨迹采集
5. 系统对轨迹做质量校验
6. 校验通过后写入本地模板库
7. 用户可在设置页管理模板并设为默认
8. 模拟提交时优先使用默认模板
9. 若无默认模板，则降级使用当前内置校园路线

### 6.2 推荐阶段

建议分三期推进：

#### Phase 1：模板落库与基础管理

- 增加路线模板数据模型
- 增加本地存储
- 增加路线模拟设置页入口
- 增加独立模板录制页
- 支持列表展示、重命名、删除、设为默认

完成后，项目即可具备“路线复用”的基本闭环。

#### Phase 2：模板驱动的模拟生成

- 模拟提交优先读取默认模板
- 根据目标距离做截取或循环拼接
- 按线段长度插值生成轨迹点
- 加入轻微随机偏移

完成后，模拟轨迹的真实性会明显优于当前版本。

#### Phase 3：质量控制与体验补强

- 增加模板质量标记
- 增加校园范围校验
- 增加模板详情统计
- 增加模板详情轨迹预览
- 增加模板来源说明与最近使用时间

完成后，路线模拟能力可以认为达到正式版预期。

## 7. 数据结构设计

建议在 `feature/running/domain` 下新增模板相关模型。

```kotlin
data class RouteTemplate(
    val id: String,
    val name: String,
    val createdAtMillis: Long,
    val updatedAtMillis: Long,
    val totalDistanceMeters: Double,
    val durationSeconds: Long,
    val pointCount: Int,
    val source: RouteTemplateSource,
    val isDefault: Boolean,
    val qualityStatus: RouteQualityStatus,
    val qualityMessage: String?,
    val points: List<TrackPoint>,
)

enum class RouteTemplateSource {
    TEMPLATE_RECORDING,
    IMPORTED,
}

enum class RouteQualityStatus {
    PASS,
    WARNING,
    REJECTED,
}
```

补充一个模拟生成输入模型，避免继续把所有逻辑塞进 `SimConfig`：

```kotlin
data class SimRouteRequest(
    val distanceKm: Double,
    val durationMinutes: Int,
    val startTimeMillis: Long,
    val preferredTemplateId: String? = null,
)
```

## 8. 存储方案

本期推荐使用 `DataStore + JSON` 或 `Room`。

建议结论：

- 若模板数量预期较少，先用 `DataStore + JSON`
- 若后续要做筛选、排序、统计，直接上 `Room`

考虑到路线模板包含大量轨迹点，建议直接使用 `Room`，理由如下：

- 模板列表与模板详情读取更清晰
- 便于后续增加字段
- 后续扩展最近使用时间、质量状态、标签时成本更低

建议表结构：

- `route_templates`
- `route_template_points`

示意：

```kotlin
@Entity(tableName = "route_templates")
data class RouteTemplateEntity(
    @PrimaryKey val id: String,
    val name: String,
    val createdAtMillis: Long,
    val updatedAtMillis: Long,
    val totalDistanceMeters: Double,
    val durationSeconds: Long,
    val pointCount: Int,
    val source: String,
    val isDefault: Boolean,
    val qualityStatus: String,
    val qualityMessage: String?,
)

@Entity(tableName = "route_template_points")
data class RouteTemplatePointEntity(
    @PrimaryKey(autoGenerate = true) val localId: Long = 0,
    val templateId: String,
    val seq: Int,
    val latitude: Double,
    val longitude: Double,
    val timestampMillis: Long,
)
```

## 9. 质量校验规则

本期不要做复杂评分，只做硬性准入和轻量提示。

### 9.1 拒绝保存条件

- 轨迹点少于 20 个
- 总长度小于 800 米
- 有效移动段过少
- 全程坐标几乎重叠，明显不是正常路线

### 9.2 警告但允许保存

- 单段漂移偏大
- 路线局部跳点较多
- 时长分布不均匀

### 9.3 校园范围校验

本期可采用简化方案：

- 先配置校园边界框 `minLat/maxLat/minLng/maxLng`
- 若超过一定比例的点落在边界框外，则标记异常

不建议第一版就做复杂多边形围栏。

## 10. 轨迹生成方案

正式版的关键升级点，是把当前“循环点位”改成“沿真实路线 polyline 取点”。

### 10.1 生成流程

1. 读取默认模板或用户指定模板
2. 对模板轨迹按顺序构建 polyline
3. 计算每一段线段长度
4. 根据目标距离决定截取还是循环拼接
5. 依据目标总时长分配每个点的时间戳
6. 沿路径插值生成目标点
7. 加入轻微抖动
8. 输出 `List<TrackPoint>`

### 10.2 截取策略

- 若模板长度大于目标距离，直接按目标距离截取前 N 米
- 可选加入起点偏移，使每次不必都从模板第一个点开始

### 10.3 拼接策略

- 若模板长度小于目标距离，则按整条路线循环拼接
- 为避免首尾过于突兀，优先选择闭环或近闭环模板

### 10.4 插值策略

- 目标点间距控制在 20-40 米
- 以线段长度为权重分配采样点数量
- 时间戳按全程时长线性分配即可，第一版不做复杂速度曲线

### 10.5 抖动策略

- 偏移控制在 2-4 米
- 不要对每个点独立大幅随机，避免锯齿感
- 优先采用低幅平滑噪声，而不是完全随机跳动

## 11. 页面与交互设计

### 11.1 入口位置

建议在“我的”页新增：

- 路线模拟

不建议直接放进“智慧运动”首页，原因如下：

- 路线模板本质上是个人配置与长期资产
- 跑步首页应继续保持“开始跑步 / 模拟提交”的单一主链
- 将路线管理塞进跑步主页面，会增加主链复杂度

点击后进入“路线模拟设置”页，建议展示：

- 默认路线
- 模板数量
- 最近录制时间
- 进入模板管理
- 录制新模板

### 11.2 模板管理页

列表项建议展示：

- 模板名称
- 总长度
- 录制时间
- 点数量
- 质量状态
- 是否默认

支持操作：

- 查看详情
- 设为默认
- 重命名
- 删除

### 11.3 模板录制页

录制模板不复用真实跑步结果页，而是独立成页。

第一版建议继续沿用当前数据仪表盘风格，展示：

- 当前状态
- 距离
- 时长
- 采集点数量
- 定位状态
- 开始录制 / 结束录制 / 保存模板

这样做的原因：

- 保持真实跑步主链简洁
- 用户能明确区分“正常跑步”和“录制模板”
- 不需要为了模板系统先重做整套地图跑步 UI

若质量不通过，则弹出明确原因，而不是静默失败。

### 11.4 模板详情页

模板详情页建议尽早补轨迹预览能力。

第一版可以是只读地图预览，不需要提供编辑能力，页面建议展示：

- 模板名称
- 总长度
- 录制时间
- 点数量
- 质量状态
- 轨迹预览

结论上，地图优先放在“模板详情查看”，而不是“主跑步录制页重构”。

### 11.5 模拟提交页

在现有参数区上方增加：

- 当前路线来源
- 默认模板名称
- 无模板时的降级提示

必要时可加：

- 本次使用默认模板
- 手动切换模板

但第一版可以先只使用默认模板，不增加复杂选择器。

## 13. 兼容与降级策略

为了不影响当前可用链路，必须保留降级方案。

### 13.1 无模板时

- 继续使用当前内置校园路线

### 13.2 默认模板失效时

- 自动回退到内置路线
- 给出轻提示，不阻断提交流程

### 13.3 模板质量为警告时

- 允许继续使用
- 在管理页显示提示标签

## 14. 验收标准

满足以下条件后，可以认为“路线模拟正式版第一阶段可交付”：

- 用户能在“我的”页进入路线模拟设置
- 用户能独立录制并保存本地路线模板
- 用户能在模板页查看、重命名、删除模板
- 用户能设置默认模板
- 模拟提交时会优先使用默认模板
- 轨迹生成不再只是固定少量点循环
- 明显异常的路线无法保存
- 用户能在模板详情页查看轨迹预览
- 没有模板时，当前模拟提交流程仍然可用

## 15. 开发顺序建议

建议按以下顺序推进：

1. 先完成模板数据结构和本地存储
2. 再完成“我的”页入口和路线模拟设置页
3. 再完成独立模板录制页与模板管理页
4. 再补模板详情轨迹预览
5. 最后替换模拟轨迹生成逻辑

这样做的原因很简单：

- 先有模板，后谈模板驱动模拟
- 先把设置和录制入口理顺，再做算法替换
- 先保证可维护，再补体验细节

## 16. 最终建议

如果只做一个“投入产出比最高”的版本，建议以以下组合作为正式版第一目标：

- “我的”页路线模拟入口
- 独立模板录制
- 模板列表管理
- 模板详情轨迹预览
- 默认模板
- 模板驱动模拟生成
- 基础质量校验

这几项完成后，路线模拟能力就已经从“能用的 MVP”升级到“符合正式版方向的可交付版本”。
